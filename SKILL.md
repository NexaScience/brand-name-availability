# ブランド名 空き状況 並列チェック

サービス名の候補を、**商標・ドメイン・SNS・既存サービス・語感(他言語の悪い意味)・発音品質**の6軸で調べ、候補ごとに表にする。

### 大原則: 人間が選ぶ“前”に調査を終える（手戻り防止）

**人間が気に入って選んだ後に「全部ドメイン取得不可/商標衝突」だと手戻りが最大になる。**
だから順序は必ず **生成 → 自動で空き調査 → 生き残りだけ人間に提示 → 人間が選ぶ**。
人間には「**調査済みで生き残った候補**」だけを見せる。候補数が多くても全部こちらで先に潰す。

### 進め方: 2段ファネル（コスト最小で全候補を篩う）

調査軸にはコスト差がある（無料即時のもの〜レート制限/規約ありのもの）。全候補に重い軸を当てると遅く高コスト。
そこで**安い軸で広く篩い、生き残りだけ深掘り**する:

1. **段1: 高速スクリーニング（全候補 15〜25件に）** — 安く速く自動化できる軸だけ:
   - **ドメイン**(RDAP=無料・即時) … 取りたいTLDが全滅なら脱落
   - **米国商標**(RapidAPI=1候補1コール) … 関係区分にLive完全一致なら脱落
   - **実在サービス**(WebSearch) … 同分野に同名の実在サービスがあれば脱落
   - ここで**明らかな✕を落とす**。残り＝候補プール（目安5〜10件）。
2. **段2: 深掘り（生き残りのみ）** — 重い/レート制限ありの軸を残った候補だけに:
   - **日本商標**(J-PlatPatブラウザDOM読み・少量) / **SNSハンドル** / **語感ネガティブ** / **発音品質**
3. **段3: 人間に提示** — 生き残りを6軸の表で提示し、**人間がこの中から選ぶ**。
   - 全候補が段1で全滅したら、**人間に選ばせず生成スキルへ差し戻して候補を作り直す**。

> 段1で重い軸（J-PlatPat等）を全候補に当てない。これが速度・コスト・規約順守の鍵。

## 入力を確認する

ユーザー/前段スキルから以下が揃っていなければ確認（揃っていれば聞かない）:
- **候補名**（生成スキルから来た**全候補**。英字＋読みカナがあると商標の称呼検索が効く）
- **ドメインのTLD**（既定: `.ai` `.io` `.com`）
- **事業分野**（商標の「区分=類」を絞るため。AIサービスなら通常 **第9類**=ソフト, **第42類**=SaaS/技術, **第35類**=広告/業務支援）

## やり方: サブエージェントで並列に投げる

**各段の中では並列**にサブエージェント（Agentツール）を投げる。各エージェントには「候補名リスト」「対象TLD/区分」と、
担当セクションの手順をそのまま渡す。ブラウザ操作するエージェントは **自分専用のタブを `tabs_create_mcp` で作り**、
`tabId` を指定して他タブを壊さないこと。

| # | 担当 | 何を調べる | 主な手段 | 段 |
|---|---|---|---|---|
| 3 | ドメイン | .ai/.io/.com の空き＋価格 | RDAP/WHOIS＋お名前.com | **段1(全候補)** |
| 2 | 米国商標 | USPTO で同一・類似の Live/Dead | API(キーあれば)/ブラウザ | **段1(全候補)** |
| 4b | 実在サービス | 同分野に同名サービスが動いていないか | WebSearch | **段1(全候補)** |
| 1 | 日本商標 | J-PlatPat で同一・類似の登録/出願 | ブラウザ(DOMテキスト) | 段2(生き残り) |
| 4a | SNSハンドル | X/IG/GitHub/YouTube のハンドル空き | WebFetch | 段2(生き残り) |
| 5 | 語感(ネガティブ) | 他言語/他地域で悪い意味に一致・酷似しないか | 多言語LLM＋WebSearch | 段2(生き残り) |
| 6 | 発音品質 | 読みやすさ・初見での発音認識・音の美しさ・人工感 | 多言語LLM＋音声合成 | 段2(生き残り) |

- **段1**: 上3軸を全候補に並列実行 → 明らかな✕を落とす（軽い・速い・規約安全）。
- **段2**: 生き残り候補にだけ残り4軸を並列実行（重い/レート制限ありを少数に限定）。
- 結果を1つの表に統合し、**生き残りを人間に提示**（→「出力」）。

### データ取得の鉄則（精度のため）

- **ブラウザを使う場合も「画像(スクショ)」で読まない。** `mcp__Claude_in_Chrome__get_page_text` か `read_page`(アクセシビリティツリー)で、商標名・ステータス・区分を**文字列として正確に**取る。OCR/目視は誤読の元。
- **構造化データ(API)が使えるものは必ずAPIを優先**（ドメインのRDAP、米国商標のAPI等）。ブラウザはAPIが無い／キーが無いときのフォールバック。
- **商標の正規API事情（2026年時点の事実）**:
  - 米国: 公式の無料フリーテキスト名称検索APIは**無い**。第三者API＋公式TSDRで自動化可能（→§2）。
  - 日本: 公式のフリーテキスト名称検索APIは**無い**。特許庁公式APIは番号指定のみ。J-PlatPat/WIPOは規約で自動クエリ禁止。よって少数候補の事前確認は**ブラウザのDOMテキスト読み（対話的・少量アクセス）**が現実解（→§1）。

## 1. 日本商標 — J-PlatPat（公式APIなし → ブラウザDOM読み）

> 公式のフリーテキスト名称検索APIは存在しない。J-PlatPat/WIPOは規約で自動クエリ（ロボット・大量DL）を禁止しているので、
> **少数候補を対話的に・少量だけ**確認する用途に限る。画像でなく**DOMテキスト**で読む。

- サイト: https://www.j-platpat.inpit.go.jp/t0100 （商標を探す・簡易検索）
- 手順:
  1. ブラウザで検索ボックスに候補名（英字とカナ読みの両方）を入れて検索。**称呼（読み）検索**が類似ヒットに強い。
  2. `get_page_text`/`read_page` で結果一覧を**テキストとして**取得。各件の **ステータス**（登録 / 出願中 / 拒絶・抹消・存続期間満了）と **区分（類）** を確認。
  3. 事業に関係する区分（AIなら 9 / 42 / 35）に絞って見る。
- 補助: 横断確認したいときは **WIPO Global Brand Database**（日本含む）もブラウザで可。ただし同様に自動クエリ禁止＝少量・対話的に。
- 判定:
  - 同一・類似名が関係区分に **「登録」または「出願中」** → ✕（要注意）
  - 別分野のみ / すべて Dead（抹消・満了・拒絶確定） / 該当なし → ○寄り

## 2. 米国商標 — USPTO（APIキーがあればAPI優先 / 無ければブラウザDOM読み）

公式に無料のフリーテキスト名称検索APIは無いが、自動化したいなら第三者API＋公式TSDRの2段が使える。

- **方法A: API（推奨・要キー）**
  - キーは `secrets/.env`に設定済み:
    - `RAPIDAPI_KEY` … RapidAPIのx-rapidapi-key
    - `RAPIDAPI_USPTO_HOST` … `uspto-trademark.p.rapidapi.com`
  - 読み込み: `set -a; source <.envパス>; set +a`
  - **名称フリーテキスト検索**（`active`=Liveのみ / `all`=Dead含む。無料Basicは250件/月・上限250件返る）:
    ```bash
    curl -s --max-time 30 \
      --url "https://${RAPIDAPI_USPTO_HOST}/v1/trademarkSearch/<name>/active" \
      -H "x-rapidapi-host: ${RAPIDAPI_USPTO_HOST}" \
      -H "x-rapidapi-key: ${RAPIDAPI_KEY}"
    ```
  - 返るJSON: `count` と `items[]`。各itemに `keyword` / `serial_number` / **`status_label`(例 "Live/Registered")** / `status_definition` / `filing_date` / `registration_date` / `expiration_date` / `description`(商品・役務) など。
  - 疎通だけ見るなら `/v1/databaseStatus`。バッチは `/v1/batchTrademarkSearch`（複数候補を一度に）。
  - **区分(International Class)を厳密に確定したい場合のみ**、得た `serial_number` で公式TSDR等に当たる（任意。事前スクリーニングは上記のstatus_label＋descriptionで十分なことが多い）。
  - キー切れ・未設定・429(レート超過)なら方法Bへ。
- **方法B: ブラウザ（フォールバック）**
  1. https://tmsearch.uspto.gov/ でキーワード検索。
  2. `get_page_text`/`read_page` で結果を**テキスト取得**し、各件の **Live / Dead** と **International Class（009 / 042 / 035 等）** を確認。
- 称呼・綴り違い（例: Nexa / Nexus / Nexar）も軽く見る。
- 判定: 関係クラスに **Live** の同一・類似 → ✕。Dead のみ / 該当なし → ○寄り。

## 3. ドメイン — .ai / .io / .com

- まず機械的に空きを判定（速い）:
  - RDAP: `https://rdap.org/domain/<name>.io`（`.com` も可）。**404=空き / 200=登録済み**。
  - `.ai` は RDAP/WHOIS が不安定なので **お名前.com で確認**するのが確実。
- 価格と最終確認: お名前.com トップ https://www.onamae.com/ の検索ボックスに候補名を入れて検索 → `.ai/.io/.com` の ○×と **価格** を見る。
- 注意: 短い `.ai/.io` は**空きでも「プレミアム」で高額**なことがある。必ず価格を記録する。
- 判定: 取りたいTLDが普通価格で空き → ○ / プレミアム高額 → △ / 登録済み → ✕

## 4. SNSハンドル・実在サービス

- SNSハンドル（`<handle>` に候補名）。WebFetch で各URLを取得し存在を判定:
  - X: `https://x.com/<handle>`
  - Instagram: `https://www.instagram.com/<handle>/`
  - GitHub: `https://github.com/<handle>`
  - YouTube: `https://www.youtube.com/@<handle>`
  - 「ページなし/アカウントが存在しません」系 → 空き ○ / プロフィールが出る → 使用中 ✕
- 実在サービス（商標未登録でも同名が動いていると不正競争リスク）。WebSearch:
  - `"<候補名>" AI` / `"<候補名>" app` / `"<候補名>" SaaS`
  - Product Hunt・GitHub・App Store/Google Play に同名・同分野があるか。
- 判定: 同分野に実在サービス → ✕ / 無名・別分野のみ → ○

## 5. 語感ネガティブチェック（他言語・他地域での悪い意味）

候補名の**発音・綴り**が、他の言語や地域で侮蔑語・卑語・縁起の悪い語・既存ブランドの悪評などに
**一致、または酷似していないか**を確認する。完全一致だけでなく、**音が似ている（1〜2音の違い）**ものも拾う。

- 対象は最低でも主要言語をカバー: 英・西・仏・独・伊・葡・中（北京/広東）・韓・日・露・アラビア語・ヒンディー。
  進出予定地域があれば優先的に追加する。
- 手段（両方やる）:
  1. **多言語LLMに問う**: `mcp__local-llms__ask`（無ければ WebSearch/自分の知識）で
     「"<候補名>" という語（およびその発音）が、各言語・各地域で否定的・侮蔑的・卑猥・不吉な意味、
     または有名な悪い語に一致もしくは酷似しないか。音写・空耳レベルの近さも含めて挙げよ」と聞く。
  2. **WebSearch で裏取り**: `"<候補名>" meaning slang` / `"<候補名>" offensive` /
     `<候補名> 意味 スラング` 等。実例・用例が出るか確認する。
- 似ているかの目安: 母音1つ違い・子音1つ違い・語末だけ違う、等は「酷似」とみなし△以上にする。
- 判定:
  - どこかの対象言語で**悪い意味に一致** → ✕
  - **酷似**（空耳で悪い語に聞こえうる） → △（要注意。ネイティブ確認を推奨）
  - 目立った問題なし → ○
- 出力では、ヒットした**言語・元の語・意味・一致/酷似の別**を必ず添える。

## 6. 発音品質チェック（読みやすさ・音の良さ）

ブランド名として「**初見で読めて、口に出しやすく、耳に心地よい**」かを評価する。
ここは主観も入るので、複数言語の話者視点と、できれば実際の音で確かめる。

- 見る観点（各5点で採点し、低い項目は理由を添える）:
  1. **読みやすさ** — 初見の綴りで迷わず読めるか。複数の読み方に割れないか。
  2. **発音認識のしやすさ** — 英語話者・日本語話者が聞いて正しく綴りを復元できるか（口頭でドメインを伝えられるか）。
  3. **音の美しさ / 言いやすさ** — 子音連続が過剰でないか、語呂・リズムが良いか、覚えやすいか。
  4. **人工感** — 母音が無く機械的すぎる、ランダム文字列的で意味の取っ掛かりが無い、等になっていないか。
- 手段:
  1. **多言語LLMに問う**: `mcp__local-llms__ask`（無ければ自分で評価）。
     「"<候補名>" を英語話者・日本語話者がそれぞれどう発音するか。読み方が割れないか。
     口頭で聞いてスペルを復元できるか。語感は自然か人工的か。上記4観点を各5点で採点し短評を付けよ」。
     **複数モデルで割れを見たいときは `mcp__local-llms__ask_multi`** を使う。
  2. **実際の音で確認（任意・推奨）**: macOS の `say` で発音を聞く。例:
     `say -v Samantha "<name>"`（英語）, `say -v Kyoko "<name>"`（日本語）。
     聞いて違和感（読み割れ・言いにくさ）があれば記録する。
- 判定: 平均が高く読み割れなし → ○ / どれか1観点が弱い → △ / 読めない・言いにくい・人工的すぎ → ✕

## 出力（＝人間への提示。ここで初めて人間が選ぶ）

**生き残った候補だけ**を、候補×軸のマトリクスで提示する。○=問題なし / △=要注意 / ✕=障害あり。
各セルは根拠を一言添える（区分・ステータス・価格・URLなど）。**段1で全滅した候補は別に「脱落理由」を簡潔に列挙**
（なぜ落ちたか分かると人間が方針を判断できる）。

```
■ 生き残り候補（この中から選んでください）
| 候補名 | 日本商標(9/42/35) | 米国商標 | .ai | .io | .com | SNS | 実在サービス | 語感(他言語) | 発音品質 | 総合 |
|--------|------------------|---------|-----|-----|------|-----|-------------|-------------|---------|------|
| Foo    | ○ 該当なし        | △ 類似Live(42) | ○ ¥xxx | ✕ 取得済 | △ premium | ○ | ○ | △ 西語"fuo"に酷似 | ○ 読みやすい | △ |

■ 段1で脱落（参考）
| 候補 | 脱落理由 |
|------|---------|
| Bar  | .ai/.io/.com 全て取得済 |
```

提示後に人間へ促す:

> 上の**生き残り候補**から、本命を選んでください。選ばれた候補は最終確認（弁理士）に進めます。
> （生き残りが0なら、生成スキルに戻って候補を作り直します。）

最後に必ず一文添える:

> これは事前スクリーニングです。類似商標の最終判断・区分選定・出願は弁理士にご確認ください。

## 注意

- ブラウザ各サイト（J-PlatPat・USPTO）は JS が重い SPA。読み込み待ち→`read_page`で要素確認→入力→**`get_page_text`で結果をテキスト取得**、の順で確実に。スクショ目視は誤読の元なので使わない。
- 候補が多い場合は、**候補ではなく「軸」で並列化**（商標JP担当が全候補をまとめて処理）すると Chrome のタブ衝突が起きにくい。
- 商標の「類似」は綴り違い・称呼（読み）が肝。完全一致だけで「クリア」と判断しない。
