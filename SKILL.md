# ブランド名 空き状況 並列チェック

サービス名の候補を、**名前の衝突・語感・発音・ドメイン・米国商標・SNS・日本商標**の7軸で調べ、候補ごとに表にする。

### 大原則: 人間が選ぶ“前”に調査を終える（手戻り防止）

**人間が気に入って選んだ後に「全部ドメイン取得不可/商標衝突」だと手戻りが最大になる。**
だから順序は必ず **生成 → 自動で空き調査 → 生き残りだけ人間に提示 → 人間が選ぶ**。
人間には「**調査済みで生き残った候補**」だけを見せる。候補数が多くても全部こちらで先に潰す。

### 大原則: 段3まで自律完走する（段の途中で確認質問をしない）

**このスキルの目的は、ユーザーとの往復（コミュニケーション）を無くすこと。**
候補を受け取ったら（**1件でも複数でも**）、段1→段2→段3（日本商標J-PlatPat）まで**自律で完走してから結果を提示する**。

- **段の途中で「段3も回しますか?」「次に進めますか?」等の確認質問をしない。** 勝手に進める。確認質問を挟むこと自体がこのスキルの失敗。
- **段1がCaution/NGでも止めない。** 最終提示の完全性のため**段3まで当てる**か、少なくとも**なぜ段3を省いたか（例: 段1で既にブロッカー確定）を明記**する。曖昧に止めない。
- 入力が決定的に不足している（候補名そのものが無い等）場合のみ、調査開始前に一度だけ確認してよい。**調査が始まったら段の境界で止まらない。**

### 進め方: 3段ファネル（安く効く順に篩う）

**番号＝実行順（§1〜§7）。上から順にやる。** 一番候補を削る&好みに直結する「音・読み・名前被り」を最初に、重い日本商標を最後に。

- **段1: 音・読み・名前被り（全候補・Web検索だけ＝安い）** — まずここで大きく削る:
  - **§1 名前の衝突**: 素の検索＋**企業系クエリ**(company/fund/capital…で実在企業の同名を炙る)＋`"<名>" AI`＋**サウンドアライク**(近綴り/同音の変種7〜12個。Skyford→Skyfire/Skyflow型)＋**素の総合リスク評価（自由探索）**
  - **§2 語感ネガティブ**: 他言語で悪い意味に一致・酷似しないか
  - **§3 発音(読みやすさ)**: 一発で読めるか・聞いて綴れるか・子音クラスタ・音節（**美しさ・好みは判定せず人間へ**）
  - → **音が悪い/読みにくい/第一印象が占有/名前被り**を落とす。残り＝音と名前が良い候補。
- **段2: 機械チェック（段1生き残りのみ・構造化データ）**:
  - **§4 ドメイン**(RDAP/whois・`.io`はDNS判定) / **§5 米国商標**(RapidAPI) / **§6 SNSハンドル**(WebFetch)
- **段3: 日本商標（最後・少数のみ・重い）**:
  - **§7 日本商標**(J-PlatPatブラウザDOM読み)。規約上も少量・対話的に。
- **提示**: 生き残りを表で提示し脱落理由も添える。**人間が選ぶのは最後**。
  - 全候補が途中で全滅したら、**人間に選ばせず生成スキルへ差し戻し**。

> 重い軸（§7 J-PlatPatブラウザ）を全候補に当てない。安いWeb検索軸(段1)で先に削るのが速度・コスト・規約順守の鍵。

## 入力を確認する

ユーザー/前段スキルから以下が揃っていなければ確認（揃っていれば聞かない）:
- **候補名**（生成スキルから来た**全候補**。英字＋読みカナがあると商標の称呼検索が効く）
- **ドメインのTLD**（既定: `.ai` `.io` `.com`）
- **事業分野**（商標の「区分=類」を絞るため。AIサービスなら通常 **第9類**=ソフト, **第42類**=SaaS/技術, **第35類**=広告/業務支援）

## やり方: サブエージェントで並列に投げる

**各段の中では並列**にサブエージェント（Agentツール）を投げる。各エージェントには「候補名リスト」「対象TLD/区分」と、
担当セクションの手順をそのまま渡す。ブラウザ操作するエージェントは **自分専用のタブを `tabs_create_mcp` で作り**、
`tabId` を指定して他タブを壊さないこと。

| § | 担当 | 何を調べる | 主な手段 | 段 |
|---|---|---|---|---|
| 1 | 名前の衝突＋**サウンドアライク**＋**企業系**＋**自由探索** | 素の検索＋企業系クエリ(実在企業/ファンドの同名)＋同名＋音が似た/綴りが近いAI・システム系(近綴り変種7〜12個)＋素の総合リスク評価 | WebSearch(＋自由探索エージェント) | **段1(全候補)・必須** |
| 2 | 語感(ネガティブ) | 他言語/他地域で悪い意味に一致・酷似しないか | 自分の多言語知識＋WebSearch裏取り | **段1(全候補)** |
| 3 | 発音(読みやすさ) | 読み割れ・聞き取り綴り復元・子音クラスタ・音節（**美しさは判定せず人間へ**） | 綴り→音の規則判定 | **段1(全候補)** |
| 4 | ドメイン | .ai/.io/.com の空き＋価格 | RDAP/WHOIS(.ioはDNS)＋お名前.com | 段2(生き残り) |

- **段2**: 段1生き残りに §4〜§6（ドメイン・米国商標・SNS）。
- **段3**: 段2生き残りにだけ §7（日本商標。重い・規約上も少量）。
- 結果を統合し、**生き残りを人間に提示**（→「出力」）。

### データ取得の鉄則（精度のため）

- **ブラウザを使う場合も「画像(スクショ)」で読まない。** `mcp__Claude_in_Chrome__get_page_text` か `read_page`(アクセシビリティツリー)で、商標名・ステータス・区分を**文字列として正確に**取る。OCR/目視は誤読の元。
- **構造化データ(API)が使えるものは必ずAPIを優先**（ドメインのRDAP、米国商標のAPI等）。ブラウザはAPIが無い／キーが無いときのフォールバック。
- **商標の正規API事情（2026年時点の事実）**:
  - 米国: 公式の無料フリーテキスト名称検索APIは**無い**。第三者API＋公式TSDRで自動化可能（→§5）。
  - 日本: 公式のフリーテキスト名称検索APIは**無い**。特許庁公式APIは番号指定のみ。J-PlatPat/WIPOは規約で自動クエリ禁止。よって少数候補の事前確認は**ブラウザのDOMテキスト読み（対話的・少量アクセス）**が現実解（→§7）。

# 段1: 音・読み・名前被り（全候補・安いWeb検索）

## 1. 名前の衝突 — 素の検索＋企業系＋AI同名＋サウンドアライク＋自由探索（段1・必須）

> **最重要**: 「同名（完全一致）が無い」だけで通してはいけない。**音が似た／綴りが近いAI・システム系サービス**こそが
> velder→velerin 型の没要因。これを段1で全候補に必ず当てる（過去に Skyford が Skyfire/Skyflow を見落とされ生き残った失敗あり）。

手順（この順で全部やる）:

0. **素の検索（まず最初に・必須）**: 候補名を**そのまま単体で**検索する — `<候補名>` だけ、`<候補名> 意味`、`<候補名> meaning`。
   AI等で限定せず、**世間で既に何に占有されているか／第一印象**を見る。
   - **中立な“元の意味”が出るのはOK＝失格にしない**: モノ(ランプ/布)・小さな地名・現象などの中立な意味がSERPに出るだけなら問題なし
     （文脈がAIから遠く、世界的に超有名でなければ、むしろ覚えやすさの利点）。
   - **ただし「多発ブランドの密度」は見る**: 中立な意味でも、**同名ブランドが複数の異分野（飲食/照明/アパレル/フォント等）で多発**しているなら、
     その密度自体を△要因とする。「個々は無関係＝OK」で済ませず、**何件の独立ブランドが同名を使っているか**を数える（多いほど混同・SEO埋没が増す）。
   - **失格(△以上)にするのは次の3つ**: ①**AI/テック/SaaS/競合になりうる企業・ブランド**がヒット、
     ②**世界的に超有名で強く占有された語**（Amazon, Sahara, Tokyo 級）で、自社名が埋もれる/混同される、
     ③**分野を問わず検索結果を占有する実在企業/ファンド/上場・被買収企業の同名**（規制業＝金融/資源・エネルギーは特に注意。下の「企業系クエリ」で必ず確認）。
   - 「あ、これは既にAIの〇〇だ」「これは誰でも知る超有名〇〇だ」「これは実在の会社/ファンドだ」という衝突を取りこぼさない。
0b. **企業系クエリ（必須・素検索で打ち切らない）**: 中立な意味（地名/モノ）が出ても**そこで止めず**、必ず次を当てて「実在企業/事業体」の同名を炙り出す:
   `"<候補>" company` `"<候補>" Ltd` `"<候補>" LLP` `"<候補>" Inc` `"<候補>" capital` `"<候補>" fund` `"<候補>" partners` `"<候補>" group` `"<候補>" asset management` `"<候補>" ventures`
   - **判定**: 規制業（金融/FCA/SEC・資産運用）・上場/被買収・報道露出の大きい同名「実在企業」がヒットしたら、**AI/テックでなくても△以上**（混同・SEO埋没・将来の越境リスク）。
   - **金融とエネルギー(石油/資源)は社名露出が大きく検索結果を占有しがち**＝特に注意。Zennor級（同名企業がドメインまで取得済み）の見落としはここで塞ぐ。
1. **完全一致（AI/テック限定）**: `"<候補名>" AI` / `"<候補名>" software` / `"<候補名>" SaaS` / `"<候補名>.ai"` / `"<候補名>" dev` を検索。
2. **サウンドアライク（必ずやる）**: 候補名から**音近似・近綴りの変種を自分で7〜12個生成**して各々検索する。生成パターン:
   - 母音1つ違い（a↔e↔o）、子音1つ違い、語末だけ違う（-o/-a/-e/-en/-er）
   - 促音・長音・ヴ↔ブの揺れ、同音異綴り（c/k, ph/f, i/y）
   - **二重子音の単純化/追加（nn↔n, ss↔s, tt↔t, ll↔l）**: 重子音は綴りを落としても同音になりやすい（例: Zennor↔Zenor、Zenner↔Zener）。両向き（増やす/減らす）で出す。
   - **発音から逆算（必ず）**: まず候補をカナ/発音に直し（例: Zennor＝ゼナー＝ZEN-er）、**同じ読みになる別綴り**を列挙する（例: Zenor/Zenner/Zenar/Xenor）。各々を `"<綴り>" AI` で検索する。
   - 検索: `"<変種>" AI` / `"<変種>.ai"` / `"<変種>" platform`
   - **語幹/接頭辞の共有群も“気づくため”に挙げてよいが、それ自体は除外理由ではない**（例 `Con-`/`Mar-`群）。
     除外するのは下の判定の「全体として取り違える」時だけ。接頭辞が被るだけのものは候補として残す。
3. **構成要素の分解チェック（必須・見落とし防止）**: 候補を意味の切れ目で分解し、**各構成要素（特に語頭の語）が、既知のAI/テックbrand名そのものと一致しないか**を確認する。
   - 例: `Glideford` = **Glide**(著名no-code/AIプラットフォーム) + ford → "Glide"を丸ごと含む。`Galewell` = **Gale**(AI広告)+well。
     これらは「相手ブランドの核を丸ごと含む」＝**除外**（"Glide系の何か"と読まれ混同する）。
   - 各構成要素を `"<要素>" AI` / `"<要素>.ai"` で引いて、**著名AIブランドと一致したら✕**。
4. 見つかった近接名は **名称・分野・音の近さ(同音/酷似/やや近い)・URL** を記録。
- 判定（**「混同テスト」で決める。厳格だが過剰除外もしない**）:
  - **核の問い**: 「その2つを声に出す/見比べて、**人は一方をもう一方と取り違えるか?**」 → **取り違える時だけ除外（✕）**。
  - **除外(✕)＝全体として混同する近接が同分野(AI/システム)にある**: ①同名、②ほぼ同音(velder↔velerin)、
    ③**相手ブランドの“核となる語”を丸ごと共有／候補の構成要素(語頭の語)が既知AIブランドそのもの**
    (Reed.aiの核=Reed を Reedmere が共有 / Bolna↔Bolsena / **Glide を Glideford が丸ごと含む / Gale を Galewell が丸ごと含む**)。
  - **❌ 過剰除外の罠（やってはいけない）**: **“ありふれた短い接頭辞が被るだけ”では除外しない**。
    例: `Con-`(Conistan↔Concentric/Conversica)、`Mar-`、`Cor-`(Cordovan↔Cordoba)、`Marl↔Mill` 等は、
    **全体が明確に別物で取り違えないので ○**。接頭辞3〜4文字の一致を理由に落とすと、中立な良候補が軒並み誤って全滅する。
  - 線引き: **“核となる語が丸ごと一致して紛らわしい”→落とす / “接頭辞・短い語片だけ共有”→落とさない**。迷う時は「実際に取り違えるか」で判断。
- **脱落理由には「どの近接名と、全体としてどう取り違えるか」を必ず書く**（接頭辞が被るだけを理由にしてはいけない）。

### 手順X: 素の総合リスク評価（自由探索・必須）

> **狙い**: 定型クエリ（上の0〜4）はチェックリストなので必ず漏れる。**「人間が普通にその名前を検索したらどう見えるか」を、ルールに縛られず自由に**調べる二重化で穴を塞ぐ（Zennor級の見落とし対策）。

- 各候補について、定型クエリとは**別に**サブエージェントを1体立て（候補ごとに**並列起動してよい**）、次をやらせる:
  - `<候補>` を**そのまま検索**し、**人間が初見で受ける印象・上位に何が出るか・どんな分野/企業/製品/人物/作品が同名で存在するか**を、**チェックリストに縛られず**列挙する。
  - 出てきた事実から、「この名前をAIサービスのブランドにした時のリスク（混同／SEO埋没／商標／語感／ネガティブ連想）」を**自分の判断で ○△✕＋理由**で評価する。
- **突き合わせ（厳しい方を採用）**: 定型チェック（手順0〜4＝企業系/サウンドアライク/分解）と自由探索（手順X）の**両方**を見て、
  **どちらか一方でも △/✕ を出したら、その候補は生存から外す**。両方○のときだけ段2へ進める。

## 2. 語感ネガティブチェック（他言語・他地域での悪い意味）

候補名の**発音・綴り**が、他の言語や地域で侮蔑語・卑語・縁起の悪い語・既存ブランドの悪評などに
**一致、または酷似していないか**を確認する。完全一致だけでなく、**音が似ている（1〜2音の違い）**ものも拾う。

- 対象は最低でも主要言語をカバー: 英・西・仏・独・伊・葡・中（北京/広東）・韓・日・露・アラビア語・ヒンディー。
  進出予定地域があれば優先的に追加する。
- 手段:
  1. **まず実行者自身の多言語知識で列挙**（あなた＝多言語LLM。別の小さいローカルLLMに問うのは二重で精度も落ちるので不要）。
     各対象言語で「"<候補名>" およびその発音が、否定的・侮蔑的・卑猥・不吉な意味、または有名な悪い語に一致/酷似しないか」を出す。
  2. **WebSearch で実例の裏取り（ここが価値）**: `"<候補名>" meaning slang` / `"<候補名>" offensive` /
     `<候補名> 意味 スラング` 等。**実在の用例・記事で裏が取れるか**を確認（推測でなく一次情報で）。
  - 知識が薄い言語/地域を補完したいときだけ、任意で外部LLM(`mcp__local-llms__ask` 等)を使ってよい（必須ではない）。
- 似ているかの目安: 母音1つ違い・子音1つ違い・語末だけ違う、等は「酷似」とみなし△以上にする。
- 判定:
  - どこかの対象言語で**悪い意味に一致** → ✕
  - **酷似**（空耳で悪い語に聞こえうる） → △（要注意。ネイティブ確認を推奨）
  - 目立った問題なし → ○
- 出力では、ヒットした**言語・元の語・意味・一致/酷似の別**を必ず添える。

## 3. 発音品質チェック（読みやすさ＝客観のみ。音の美しさは判定しない）

> **重要**: 「音が美しい／語呂が良い／覚えやすい」は**主観の嗜好**で、モデルの採点は実ユーザーの好みと何度も外れた。
> **だから機械では“読みやすさ”という客観軸だけ判定し、美しさ・好みは判定せず人間に委ねる**（候補に添えて人間が選ぶ）。
> `say`(音声合成)は実行者が音を聞けないので**使わない**（ハッタリ）。発音は綴り→音の規則で判断する。

- **客観軸（実行者が綴りの規則で判定。○△✕を付けるのはここだけ）**:
  1. **読み割れ** — 英語話者の綴り→読みが**複数に分岐**しないか（例: Chine→チャイン/シャイン/チーン、Olen→Owen/Olin）。分岐するなら△以上。
  2. **聞き取り→綴り復元** — 口頭で言われて綴りを一意に戻せるか（同音異綴りが多い/ヴ等の希少音は△）。
  3. **子音クラスタ** — 連続子音や言いにくい結合(`Lj-`,`sh-w`,`ndv`等)が無いか。あれば△以上。
  4. **音節・母音** — 母音が無く機械的すぎないか、長すぎ(4音節超)ないか。
- **判定（客観軸のみ）**: 読み割れ無し＋子音クラスタ無し＋適度な音節 → ○ / 1つでも弱い → △ / 読めない・言いにくい → ✕。
- **美しさ・好みは採点しない**。出力に「読み(カナ)・音節数」を添え、**最終的な響きの好みは人間が選ぶ**。

# 段2: 機械チェック（段1生き残りのみ・構造化データ）

## 4. ドメイン — .ai / .io / .com

- まず機械的に空きを判定（速い）:
  - `.com` は RDAP に **`-L`（リダイレクト追従）を付ける**。rdap.org は登録局へ302リダイレクトするため、付けないと判定不能(302/000)になる:
    ```bash
    curl -s -L -o /dev/null -w "%{http_code}" --max-time 20 "https://rdap.org/domain/<name>.com"
    # 200=登録済み / 404=空き。
    ```
  - **`.io` は rdap.org が誤判定する（登録済みでも404を返す）ので使わない。** 判定は次の2段:
    ```bash
    dig +short NS <name>.io @8.8.8.8   # 一次篩い: NSあり=登録済み / 空=空きの可能性
    whois <name>.io                    # 確定: 「Domain not found/No match」=空き（NS空でも予約済みがあるのでwhoisで確定）
    ```
  - `.ai` は **whois が確実**: `whois <name>.ai` で「No match/NOT FOUND」=空き、「Domain Name/Name Server」=登録済み。
  - 短時間に多数叩くと **429（レート制限）** が出る。出たら間隔をあけて再試行（または対象を生き残りに絞る）。
  - 実装メモ: シェルは **zsh**。`for n in $list` は単語分割されないので**配列** `list=(a b c)` を使う。
- 価格と最終確認は **お名前.com** で（プレミアム価格に注意）: https://www.onamae.com/ の検索ボックスに候補名 → `.ai/.io/.com` の ○×と **価格**。
- 注意: 短い `.ai/.io` は**空きでも「プレミアム」で高額**なことがある。必ず価格を記録する。
- **証跡を残す（断定しない）**: whois の結果は **raw 出力の該当行**（"No match" / "Name Server:" 等）を**1行だけ証跡として残す**。「空いていそう」で確定しない。
  パーク/予約ドメインは Name Server が入る → **登録済み扱い**にする。
- **関門ルール（最重要）**: 第一希望TLD（**既定は `.ai` と `.io` の両方**）が**いずれも取得不可**（登録済み or 高額プレミアム）なら、
  他軸が良くても**総合は✕寄りにし、生き残りに含めない**。「`.io` だけ空き」では**両取り前提の事業要件を満たさない**ので生存させない。
- 判定: 取りたいTLD（既定 .ai と .io の両方）が普通価格で空き → ○ / 片方のみ空き・プレミアム高額 → △ / 第一希望が両方とも不可 → ✕（生存させない）

## 5. 米国商標 — USPTO（APIキーがあればAPI優先 / 無ければブラウザDOM読み）

公式に無料のフリーテキスト名称検索APIは無いが、自動化したいなら第三者API＋公式TSDRの2段が使える。

- **方法A: API（推奨・設定済み）**
  - キーは `secrets/.env`（`/Users/kumacmini/Library/CloudStorage/Dropbox/secrets/.env`）に設定済み:
    - `RAPIDAPI_KEY` … RapidAPIのx-rapidapi-key
    - `RAPIDAPI_USPTO_HOST` … `uspto-trademark.p.rapidapi.com`
  - 読み込み: `set -a; source "<上記.envパス>"; set +a`
  - **名称フリーテキスト検索**（`active`=Liveのみ / `all`=Dead含む。無料Basicは250件/月・上限250件返る）:
    ```bash
    curl -s --max-time 30 \
      --url "https://${RAPIDAPI_USPTO_HOST}/v1/trademarkSearch/<name>/active" \
      -H "x-rapidapi-host: ${RAPIDAPI_USPTO_HOST}" \
      -H "x-rapidapi-key: ${RAPIDAPI_KEY}"
    ```
  - 返るJSON: `count` と `items[]`。各itemに `keyword` / `serial_number` / **`status_label`(例 "Live/Registered")** / `status_definition` / `filing_date` / `registration_date` / `expiration_date` / `description`(商品・役務) など。
  - **重要: `count` は部分一致（例: "aron"で"ARONGIN"等）を含む。判定は `count` で見ず、`keyword` が候補名と完全一致(case-insensitive)で `status_label` に "Live" を含む件を数える**:
    ```python
    ex=[i for i in items if (i.get('keyword') or '').strip().lower()==name.lower()
        and 'live' in (i.get('status_label') or '').lower()]
    ```
  - 完全一致Liveがあっても、`description` から**関係区分(9/42/35=ソフト/SaaS/広告)か無関係区分(化学・バルブ等)かを見る**。無関係区分のみなら△、関係区分にあれば✕寄り。
  - 疎通だけ見るなら `/v1/databaseStatus`。バッチは `/v1/batchTrademarkSearch`（複数候補を一度に）。
  - **無料Basicは250件/月の上限**。全候補×綴り変種で叩くと枯れる → **段2の生き残りだけ・1候補1クエリ**に圧縮して叩く（3段ファネルと整合）。
  - **区分(International Class)を厳密に確定したい場合のみ**、得た `serial_number` で公式TSDR等に当たる（任意。事前スクリーニングは上記のstatus_label＋descriptionで十分なことが多い）。
  - キー切れ・未設定・429(レート超過)なら方法Bへ。
- **方法B: ブラウザ（フォールバック）**
  1. https://tmsearch.uspto.gov/ でキーワード検索。
  2. `get_page_text`/`read_page` で結果を**テキスト取得**し、各件の **Live / Dead** と **International Class（009 / 042 / 035 等）** を確認。
- 称呼・綴り違い（例: Nexa / Nexus / Nexar）も軽く見る。
- 判定: 関係クラスに **Live** の同一・類似 → ✕。Dead のみ / 該当なし → ○寄り。
